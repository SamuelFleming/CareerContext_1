## Dashboard Product Role

The Dashboard answers:

1. **Who am I here?** — personalised Interactive CV identity
2. **Is my core context ready?** — conditional completeness prompt
3. **What does my career narrative look like?** — Core Context summary preview inside the CV
4. **What evidence am I building?** — placeholder / future experience highlights
5. **What have I worked on recently?** — collapsible recent-evidence panel (Phase 2+)

Opportunities and documents are **out of scope** for Phase 1 dashboard data (quick-action stub only).

## Layout Contract

### Left column — Interactive CV (`interactiveCv`)

Primary card styled like a living CV (matches existing `DashboardPage` visual language).

| Section | Phase 1 behaviour | Data fields |
|---------|-------------------|-------------|
| Header | Name + headline | `identity.fullName`, `identity.headline` |
| Contact strip | Optional one-line contact | `identity.email`, `identity.mobile`, `identity.location` |
| Profile summary | Preview of raw summary or empty-state prompt | `interactiveCv.summaryPreview`, `interactiveCv.summaryUpdatedAt`, `interactiveCv.reviewSuggested` |
| Core competencies | Empty chips + “Add experiences to populate skills” | `interactiveCv.coreCompetencies[]` (empty in Phase 1) |
| Experience highlights | Empty-state or 0 cards | `interactiveCv.highlightExperiences[]` (empty in Phase 1) |

### Right column

#### A. Profile completeness (`profileCompleteness`)

- Rendered **only when** `profileCompleteness.showPrompt === true`
- Hidden when profile is sufficiently complete (`status: "ready"`)

#### B. Collapsible evidence panel (`evidencePanel`)

Single container toggling between two views:

| View | Phase 1 | Phase 2+ |
|------|---------|----------|
| `evidenceSummary` | Empty state: evidence collection comes next | Compact experience/activity counts or highlights |
| `recentActivity` | Empty state: recent updates appear here | Top 5 items by `updatedAt` (Activities + Journal entries) |

Default expanded view: `evidenceSummary`.

#### C. Quick actions (`quickActions`)

| Action | Route | Phase 1 |
|--------|-------|---------|
| Edit Core Context | `/profile` | Enabled |
| Add Experience | `/experiences` | Disabled — “Coming soon” |
| Add Opportunity | `/opportunities` | Disabled — “Coming soon” |
| Open Journal | `/journal` | Disabled — “Coming soon” |

## Profile Completeness Rules

Calculated on the **backend** in `GET /api/dashboard`.

| Field | Source | Weight | Complete when |
|-------|--------|--------|---------------|
| Full name | `CoreContext.fullName` | 15% | non-empty trimmed string |
| Headline | `CoreContext.headline` | 15% | non-empty trimmed string |
| Location | `CoreContext.location` | 10% | non-empty trimmed string |
| Career summary | `CoreContext.rawSummaryMd` | 30% | length ≥ 100 characters |
| Core resume | `User.coreResumeMd` | 30% | length ≥ 100 characters |

**Score:** sum of weights for complete fields (0–100).

**Status:**

| Score | Status |
|------:|--------|
| 0–24 | `empty` |
| 25–74 | `in_progress` |
| 75–100 | `ready` |

**`showPrompt`:** `true` when `status !== "ready"`.

**`missing`:** array of field keys still incomplete, e.g. `["rawSummaryMd", "coreResumeMd"]`.

**`nextAction`:** first missing field mapped to a label + `/profile` route.

---

## Core Context Preview Rules

Used inside the Interactive CV profile summary section.

- **Preview length:** first 250–400 characters of `rawSummaryMd`
- **Truncation:** append `…` when truncated
- **Empty:** show guided empty-state copy linking to `/profile`
- **Staleness:** `reviewSuggested: true` when `summaryUpdatedAt` is older than 30 days (UX hint only)
- **Timestamp:** display `summaryUpdatedAt` when present

---

## Core Resume Preview Rules

Separate compact card or subsection (below Interactive CV or within right column — implementation choice in **006.4**; data provided by API).

| Field | Rule |
|-------|------|
| `exists` | `coreResumeMd` trimmed length > 0 |
| `previewMd` | first 200 characters |
| `updatedAt` | `User.coreResumeUpdatedAt` |

Core Resume remains on `User` for MVP (not a `Document` entity).

## Recommended Endpoint

```
GET /api/dashboard
```

- Authenticated (`Authorization: Bearer <token>`)
- Single compact payload for Screen 4
- Does **not** replace `GET /api/profile` (profile editing stays on Screen 5)
