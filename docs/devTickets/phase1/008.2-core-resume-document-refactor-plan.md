## Current MVP Design

```js
{
  coreResumeMd: String,
  coreResumeUpdatedAt: Date
}
```

### API (implemented)

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/profile` | Returns `coreResumeMd`, `coreResumeUpdatedAt` in profile envelope |
| PUT | `/api/profile/core-resume` | Updates `coreResumeMd`; sets `coreResumeUpdatedAt` |

### UI (implemented → upgrading in 008.3 / 008.4)

- `CoreResumeEditor` on Screen 5 (Profile / Core Context)

`profileCompletenessService` treats non-empty `User.coreResumeMd` as one of five equal checks (20%).

### Document backend (not implemented)

- No `server/src/models/Document.js`
- `/api/documents/*` routes return **501**
- `Document` entity exists in `docs/05_data_model.md` and `docs/08_api_contract.md` only

## Why Core Resume Is Profile-Owned for Now

1. **Phase 1 scope** — Profile + Core Context must work without Document CRUD, Screen 12, or file upload infrastructure.
2. **Single canonical source** — MVP needs one resume string for AI workflows and completeness; versioning is not required yet.
3. **Simpler API** — `GET/PUT /api/profile` covers the only resume use case today (edit source resume on Profile).
4. **Documents serve a broader role** — Cover letters, tailored variants, evaluation reports, and opportunity-linked outputs need the full `Document` model with `type`, `status`, `opportunityId`, and source tracing — not just core resume.
5. **Avoid throwaway work** — Building `Document` only for core resume would duplicate profile endpoints and require migration before Phase 3 features exist.

**008.3** and **008.4** improve UX **without** changing this storage decision.

## Future Target Model

Align with `docs/05_data_model.md` Document entity. Field naming in data model uses `content` (Markdown/plain text); API examples may expose `contentMd` as an alias — pick one at implementation time.

```js
// Document (future)
{
  _id,
  userId,
  type: 'core_resume',      // singleton per user in practice
  title,                    // e.g. "Core resume"
  content,                  // Markdown body (canonical field name in 05_data_model)
  filePath,                 // optional — populated when sourced from upload (Phase 3+)
  status,                   // draft | final | archived
  opportunityId,            // null for core_resume
  sourceExperienceIds,      // future traceability
  sourceActivityIds,
  sourceJournalEntryIds,
  sourceEvaluationId,
  aiRunId,
  createdAt,
  updatedAt
}
```

### User after migration

```js
{
  // ...existing auth fields...
  coreResumeDocumentId,     // pointer to canonical core_resume Document
  // Deprecated after migration window:
  // coreResumeMd, coreResumeUpdatedAt — remove or keep as read-only mirror one release
}
```

### Product rule

- **One** `type: 'core_resume'` document per user (enforced in service layer).
- Tailored / opportunity-specific resumes are **separate** documents (`type: 'resume_variant'` or similar — define in DOC-001).

## Refactor Triggers

Migrate off `User.coreResumeMd` when **one or more** become true:

| Trigger | Why Document is required |
|---------|--------------------------|
| Resume **versioning** | Multiple resume revisions over time |
| **Document export** (PDF/DOCX) | Needs `filePath` or export pipeline on Document |
| **Generated targeted resumes** | Variants per opportunity, not one User field |
| **Linking documents to opportunities** | `Document.opportunityId` and workspace APIs |
| **Source evidence tracing** | `sourceExperienceIds`, `sourceActivityIds`, etc. |
| **Document history / audit** | Status transitions, archived copies |

**Do not migrate** for these alone:

- Core resume should use `MarkdownEditor` → **008.3** only
- User might upload a PDF someday → **008.4** scaffold + DOC-001; no migration yet
- “Documents feel more correct architecturally” → wait for a hard trigger

## Migration Shape (when triggered)

Prerequisite: **DOC-001** (Document CRUD) implemented and tested with at least one document type (e.g. `cover_letter`).

### Step 1 — Data migration script

For each user where `coreResumeMd` is non-empty:

1. Create `Document { userId, type: 'core_resume', title: 'Core resume', content: coreResumeMd, status: 'final' }`
2. Set `User.coreResumeDocumentId` to new document `_id`
3. Log users with empty resume (optional placeholder document or leave null)

### Step 2 — API compatibility layer

- `GET /api/profile` returns:

```json
{
  "coreResume": {
    "documentId": "...",
    "contentMd": "...",
    "updatedAt": "..."
  }
}
```

- Keep `PUT /api/profile/core-resume` as a thin wrapper updating the core_resume Document (deprecate later).
- Optionally mirror `coreResumeMd` on response for one release.

### Step 3 — Completeness service

- `profileCompletenessService` checks core resume via Document content (or `coreResumeDocumentId` presence + non-empty content).

### Step 4 — UI

- `CoreResumeEditor` loads/saves via document id or profile wrapper endpoint.
- Upload mode (**008.4**) stores file → extract/import to Document `content` + `filePath`.

### Step 5 — Cleanup

- Remove `coreResumeMd` / `coreResumeUpdatedAt` from `User` schema after deprecation window.

## Relationship to Upload Scaffold (008.4)

| Phase | Upload behaviour |
|-------|------------------|
| **008.4 (now)** | UI tab/segment for Upload; disabled or “coming with Documents”; no file persisted |
| **Phase 3** | Upload → `Document` with `filePath`; text extraction → `content` |
| **Post DOC-002** | Core resume upload updates `core_resume` Document, not `User` |

Do not implement Multer/storage for core resume until **DOC-001** exists.
